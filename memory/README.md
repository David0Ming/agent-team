# Memory 文件职责地图

> 目标：让每条新增内容都有明确落点，避免乱写、重写、重复写。

## 一、总原则

### 先判断“这条信息属于什么”
新增内容前先分 5 类：
1. **用户长期信息** → `USER.md`
2. **Agent 工作规则** → `AGENTS.md` / `SOUL.md`
3. **待办/项目状态** → `memory/projects.md` 或 `projects/.../docs/`
4. **通用经验/方法论** → `memory/lessons*.md` / `memory/learning/knowledge/`
5. **当天发生的事** → `memory/daily/YYYY-MM-DD.md`

### 默认落点顺序
- **先 daily，后升级**：当天先记到 `memory/daily/`
- **能复用再升级**：稳定规则再提炼到 `lessons*.md` / `knowledge/`
- **项目知识不进 memory 主题文件**：进对应 `projects/{project}/docs/`
- **工具配置与故障细节**：优先进 `memory/tools.md`

### 写入前四步
1. 相关读取
2. 查重
3. 合并写入
4. 写后验证

## 二、核心文件职责

### 1. `memory/daily/YYYY-MM-DD.md`
**职责**：当天发生了什么、做了什么、发现了什么。

**适合写**：
- 心跳留痕
- 当天任务进展
- 当轮讨论结论
- 临时阻塞 / 异常
- 还没提炼成熟的经验

**不适合写**：
- 永久规则
- 长期偏好
- 项目规范正文

**一句话判断**：
> 这是“今天的记录本”。

---

### 2. `memory/projects.md`
**职责**：跨天跟踪的待办、项目、安排。

**适合写**：
- 未完成事项
- 截止时间
- 项目状态变化
- 已取消 / 已完成状态

**不适合写**：
- 过程细节
- 方法论总结
- 项目内部操作说明

**一句话判断**：
> 这是“你现在还要做什么”。

---

### 3. `memory/lessons.md`
**职责**：所有 Agent 通用的工作规则。

**适合写**：
- 通用流程规则
- 诊断方法论
- 文件管理原则
- 验证与汇报原则

**不适合写**：
- DJJ 独有的统筹经验
- 技术团队专属经验
- 工具具体配置值

**一句话判断**：
> 这是“大家都适用的经验”。

---

### 4. `memory/lessons-djj.md`
**职责**：DJJ 作为统筹者的独有经验。

**适合写**：
- 任务路由
- owner 判断
- 汇报时机
- 记忆闭环 / 交付闭环
- 面试复盘框架这类“统筹判断方法”

**不适合写**：
- 网络代理配置
- Python 包安装方法
- 某工具报错排查细节
- 项目业务知识

**一句话判断**：
> 这是“DJJ 怎么做统筹”。

---

### 5. `memory/lessons-dming.md`
**职责**：技术团队通用的工程经验。

**适合写**：
- 架构设计取舍
- 调试方法
- 测试策略
- 稳定性 / 性能经验

**不适合写**：
- 用户沟通方式
- DJJ 汇报策略
- 工具账户配置

**一句话判断**：
> 这是“技术团队怎么做技术判断”。

---

### 6. `memory/lessons-david.md`
**职责**：运营、内容、传播方向的可复用经验。

**适合写**：
- 内容策略
- 传播判断
- 视频/文案复盘规则

**一句话判断**：
> 这是“运营团队怎么做内容判断”。

---

### 7. `memory/tools.md`
**职责**：工具、环境、配置、故障处理入口。

**适合写**：
- OpenClaw 配置
- 模型路由
- 代理配置
- API / 插件 / CLI 约定
- 常见错误与修复入口

**不适合写**：
- 一般工作方法论
- 用户偏好
- 项目业务规范

**一句话判断**：
> 这是“工具怎么用、环境怎么配”。

---

### 8. `memory/learning/knowledge/*.md`
**职责**：可复用知识卡片。

**适合写**：
- 某个主题的系统知识
- 经过提炼的概念、方法、适用场景
- 可反复调用的结构化认知

**不适合写**：
- 一次性任务记录
- 今天做了什么
- 只对某个项目有效的细节

**一句话判断**：
> 这是“知识库”，不是日志。

---

### 9. `memory/learning-log.md`
**职责**：学习发现的轻量留痕。

**适合写**：
- 今天学到了什么
- 某篇资料/趋势值得后续提炼

**不适合写**：
- 完整知识卡正文
- 项目待办

**一句话判断**：
> 这是“学习草稿箱”。

---

### 10. `memory/emails/` 与 `memory/email-todos.md`
**职责**：邮件原始留存与邮件待办。

**适合写**：
- 重要邮件 JSON
- 邮件触发的行动项

**一句话判断**：
> 这是“邮件证据”和“邮件待办”。

---

### 11. `memory/heartbeat-state.json` / `memory/heartbeat-status.json`
**职责**：心跳状态机文件。

**适合写**：
- 最新检查时间
- 系统状态
- 当前异常摘要

**不适合写**：
- 长篇解释
- 学习成果正文

**一句话判断**：
> 这是“机器读的状态文件”。

---

### 12. `projects/{project}/docs/`
**职责**：项目内部知识，而不是个人长期记忆。

**适合写**：
- 项目流程
- 业务规范
- 领域术语
- 项目复盘

**不适合写**：
- DJJ 个人工作方法
- 用户长期偏好

**一句话判断**：
> 项目知识归项目，不归 memory 总主题文件。

---

## 三、最常见内容怎么放

### 场景 A：今天聊完一次面试复盘
- 先写：`memory/daily/YYYY-MM-DD.md`
- 若形成稳定复盘规则：再提炼到 `memory/lessons-djj.md`
- 若是用户求职方向变化：同步 `USER.md`

### 场景 B：发现 web_search 报错与代理有关
- 配置与修复命令：`memory/tools.md`
- 通用诊断方法：`memory/learning/knowledge/knowledge-network-tools-diagnosis.md`
- 当天处理经过：`memory/daily/YYYY-MM-DD.md`

### 场景 C：新增一个长期待办
- 写：`memory/projects.md`
- 如果今天已经讨论过，也可在 `daily` 留一笔

### 场景 D：某项目的内容合规规范
- 写：`projects/{project}/docs/xxx.md`
- 不写进 `lessons-djj.md`

### 场景 E：某条规则全体 Agent 都适用
- 写：`memory/lessons.md`

### 场景 F：某条规则只影响 DJJ 的路由/汇报
- 写：`memory/lessons-djj.md`

## 四、以后新增内容的快速路由

可以直接按这张表判断：

- **这是今天发生的事吗？** → `daily/`
- **这是未完成事项吗？** → `projects.md`
- **这是所有 Agent 都适用的规则吗？** → `lessons.md`
- **这是 DJJ 独有统筹经验吗？** → `lessons-djj.md`
- **这是技术方法吗？** → `lessons-dming.md` 或 `knowledge/`
- **这是工具配置/故障吗？** → `tools.md`
- **这是项目内知识吗？** → `projects/.../docs/`
- **这是用户长期偏好吗？** → `USER.md`

## 五、当前发现的混乱点

目前 `memory/` 里还存在一些历史遗留问题：
- 顶层有一批“按日期命名但不在 daily/ 的文件”
- 一些项目/研究型文件混在 `memory/` 顶层
- `README.md` 之前更像旧索引，不像真正的职责地图

处理原则：
- **先定规则，再逐步收敛**
- 不一次性大迁移，避免误伤引用路径
- 新增内容从现在开始按这张地图执行

---

## 六、执行准则

1. **先找现有文件，再决定是否新建**
2. **同类内容优先合并，不要到处散落**
3. **先记录事件，再提炼规则**
4. **项目知识归项目，个人方法归 lessons，工具问题归 tools**
5. **写入后要能回答一句：以后谁会在什么时候读它？**
