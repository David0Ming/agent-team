## 1️⃣ 复述：这个知识是什么

**核心观点**：生产级AI Agent不是一个prompt，而是一个分布式系统，LLM只是其中的规划器/执行器。

**5个关键架构原则**：

### 1. 工具调用 = API契约
- 工具必须有类型验证（JSON Schema/Zod）
- 工具必须幂等（避免重复执行副作用）
- 工具输出必须结构化：`{ ok, data, error, meta }`
- 每个工具调用都有预算（timeout、max retries、max cost）

### 2. 显式状态机
- 分离LLM决策（概率性）和状态转换（确定性）
- 使用Reducer模式管理状态
- 状态包含：goal、constraints、plan、steps、memories、policy

### 3. 4层记忆架构
- **Layer 1 - 工作记忆**：当前思考的内容（临时）
- **Layer 2 - 对话记忆**：最近N轮对话 + 滚动摘要
- **Layer 3 - 任务记忆**：生成的文件、决策、命令（结构化）
- **Layer 4 - 长期记忆**：用户偏好、组织事实（需明确同意）

### 4. 迭代规划
- 不追求"一次完美计划"
- 浅层计划 → 执行第1步 → 根据结果重新规划
- 计划存储为JSON数据结构

### 5. 基于轨迹的评估
- 评估完整轨迹（工具选择 + 结果），不只是最终答案
- 混合指标：自动化 + 人工评判
- 在CI中集成评估

## 2️⃣ 应用：在我们系统中的应用

**当前系统分析**：
- ✅ 我们有工具调用（Read、Write、exec等）
- ⚠️ 工具输出不够结构化（缺少meta信息）
- ❌ 没有显式状态机（依赖对话历史）
- ⚠️ 记忆分层不清晰（memory文件混杂）
- ❌ 没有评估机制

**可以改进的地方**：

1. **工具输出标准化**
   - 当前：工具返回原始输出
   - 改进：包装为 `{ ok, data, error, meta: { tool, durationMs, cacheHit } }`

2. **状态管理**
   - 当前：依赖对话历史
   - 改进：引入AgentState结构，记录goal、constraints、steps

3. **记忆分层**
   - 当前：memory文件混杂
   - 改进：
     - 工作记忆 → session state
     - 对话记忆 → memory/daily/YYYY-MM-DD.md
     - 任务记忆 → memory/projects/
     - 长期记忆 → memory/identity.md, user.md等

4. **评估机制**
   - 当前：无
   - 改进：记录每次任务的轨迹，定期评估质量

## 4️⃣ 验证：我是否真正掌握

**测试问题**：
1. 为什么工具调用需要幂等性？
   - 答：避免重试时产生重复副作用（如重复扣款、重复发邮件）

2. 为什么要分离LLM决策和状态转换？
   - 答：LLM决策是概率性的（不可预测），状态转换应该是确定性的（可调试、可重放）

3. 向量数据库适合存储什么类型的记忆？
   - 答：文档/代码库上下文、长对话回忆。不适合：关键事实、权限策略、金钱交易

**实践验证**：
- ✅ 我能解释这些概念
- ⚠️ 还没有实际应用到系统中
- 📝 需要在下次开发时实践

## 5️⃣ 记忆：关键要点

**记住这3点**：
1. **Agent = 分布式系统**，不是prompt
2. **工具 = API契约**，需要验证和预算
3. **记忆 = 4层架构**，不只是向量数据库

**反模式**：
- ❌ 把所有东西都塞进prompt
- ❌ 依赖无限对话历史
- ❌ 把向量数据库当作万能记忆
