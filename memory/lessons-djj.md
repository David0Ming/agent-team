### 敏感词汇管理（2026-03-10）
**场景**：嘟嘟巴士prompt生成时发现使用了"暴力"、"欺骗"、"品牌"等敏感词汇

**原则**：
1. 任何面向公众的内容，必须避免负面情绪词（暴力、欺骗、恐惧、焦虑）
2. 避免明显的商业化词汇（品牌、广告、营销、推广）
3. 建立敏感词替代词库，保持创意的同时确保合规

**替代策略**：
- 负面词 → 中性描述（"暴力" → "对比"）
- 商业词 → 具体名称（"品牌" → "嘟嘟巴士"）

### 数据分析方法（2026-03-10）
**场景**：竞品数据分析时，用户要求"昨天单日数据"而不是累计数据

**原则**：
1. 明确数据类型：累计 vs 单日 vs 平均
2. 数据对比要同口径（都用单日或都用累计）
3. 趋势分析用时间序列，不用单点数据

## 技术问题解决经验

### 网络搜索工具诊断方法（2026-03-15）

**场景**：OpenClaw的web_search工具无法使用，需要系统诊断

**错误表现**：
- 初始错误："fetch failed"（网络连接失败）
- 配置代理后：Gemini变成"API key invalid"（网络通了）
- 换成Kimi：又回到"fetch failed"
- 换成Brave：成功！

**诊断方法（4步系统诊断法）**：

1. **工具清单**：列出所有可用工具
   - web_search（主要工具）
   - web_fetch（备用方案）
   - Codex --search（高级搜索）
   - agent-browser（浏览器自动化）

2. **逐一测试**：不要简单说"不行"
   - 测试每个工具的实际表现
   - 记录具体的错误信息
   - 区分"完全不能用"和"部分可用"

3. **根因分析**：深入挖掘问题原因
   - "fetch failed" → 网络连接问题
   - "API key invalid" → 认证问题（但网络通了）
   - 不同提供商表现不同 → 配置或兼容性问题

4. **系统解决**：从根本上解决问题
   - 配置代理环境变量
   - 测试多个提供商
   - 找到最稳定的方案

**关键教训**：
- ❌ 错误做法：遇到"fetch failed"就说"工具不能用"
- ✅ 正确做法：系统诊断，找出真正原因，逐步解决

**可复用模板**：
```
问题：[工具名]不能用
1. 列出相关工具清单
2. 逐一测试并记录错误
3. 分析错误类型（网络/配置/权限/兼容性）
4. 针对性解决（代理/配置/替代方案）
5. 验证并记录最终方案
```

### OpenClaw代理配置（2026-03-15）

**场景**：OpenClaw的web_search需要访问外网API（Gemini/Brave等）

**问题**：
- OpenClaw的内置工具不自动使用系统代理
- 需要在配置中显式设置环境变量

**解决方案**：
```bash
# 配置代理环境变量
openclaw config set env.HTTP_PROXY "http://127.0.0.1:7890"
openclaw config set env.HTTPS_PROXY "http://127.0.0.1:7890"

# 重启Gateway生效
openclaw gateway restart
```

**验证方法**：
- 测试前：错误信息从"fetch failed"变化
- Gemini：变成"API key invalid"（说明网络通了）
- Brave：完全成功

**经验**：
1. 代理配置要在应用层设置，不能只依赖系统代理
2. 不同工具对代理的支持不同，需要逐一测试
3. 错误信息的变化是诊断的关键线索

### 搜索提供商选择（2026-03-15）

**场景**：OpenClaw支持5种搜索提供商，需要选择最合适的

**提供商对比**：
1. **Brave Search**（推荐）
   - 优点：稳定、免费额度、结构化结果
   - 缺点：需要注册API密钥
   - 适用：生产环境

2. **Gemini**
   - 优点：AI合成答案、引用来源
   - 缺点：需要Google服务访问
   - 适用：需要AI理解的场景

3. **Kimi**
   - 优点：中文优化
   - 缺点：网络兼容性问题
   - 适用：纯中文场景（需要测试）

4. **Perplexity/Grok**
   - 优点：AI增强搜索
   - 缺点：需要API密钥
   - 适用：深度研究

**选择原则**：
1. 优先选择稳定性高的（Brave）
2. 测试网络兼容性
3. 考虑免费额度和成本
4. 根据使用场景选择

**经验**：
- 不要只配置一个提供商
- 准备备用方案
- 记录每个提供商的特点和限制

### 系统Python包管理（2026-03-15）

**场景**：尝试安装edge-tts时遇到系统保护

**错误信息**：
```
error: externally-managed-environment
This environment is externally managed
```

**原因**：
- Ubuntu/Debian使用PEP 668保护系统Python
- 防止pip破坏系统包管理

**正确做法**：
1. **虚拟环境**（推荐）：
   ```bash
   python3 -m venv myenv
   source myenv/bin/activate
   pip install package
   ```

2. **pipx**（CLI工具）：
   ```bash
   pipx install package
   ```

3. **用户级安装**（不推荐）：
   ```bash
   pip install --user package
   ```

**错误做法**：
- ❌ 使用--break-system-packages
- ❌ 修改系统Python配置
- ❌ 要求sudo权限

**经验**：
- 遇到系统保护时，应该遵循最佳实践
- 不要绕过安全机制
- 虚拟环境是标准做法

## 工作方法改进

### 后台任务通知机制（2026-03-16）

**场景**：使用Claude Code修复项目代码，任务完成后没有及时通知用户

**错误做法**：
- ❌ 18:08收到系统通知"Done"
- ❌ 先自己查看代码细节
- ❌ 18:11才告诉用户（延迟3分钟）
- ❌ 说要"每2-3分钟轮询报告进度"（浪费token）

**正确做法**：
1. **启动任务时**：告诉用户"Claude Code开始工作了"
2. **收到通知后**：立即告诉用户"完成了！正在查看结果..."
3. **查看后**：简要说明完成了什么

**核心原理**：
- coding-agent技能使用"openclaw system event --mode now"自动通知
- 系统收到通知后会触发我的回复
- 我应该立即转告用户，而不是先自己看完

**不要做的**：
- ❌ 不轮询后台任务（浪费token）
- ❌ 不每2-3分钟报告（浪费token）
- ❌ 不先查看完再通知（延迟反馈）

**记忆点**：
收到系统通知 = 立即告诉用户，不要先自己看

### 记忆闭环缺陷反思（2026-03-24）

**场景**：用户已经在聊天窗口中完成过重要复盘（如面试复盘），但后续我在检索时只查文件，未能识别出“聊过但没落盘”的内容，导致我错误地把“没有正式文件”近似成“没有复盘过”。

**问题本质**：
1. 把“对话里讨论过”误当成“已经沉淀”
2. 任务完成后没有强制做记忆闭环
3. 心跳流程偏巡检，不会自动把最近聊天中的关键成果抽取到长期记忆
4. 检索时过度依赖文件命中，没有明确区分“无记录”与“未落盘”

**严重性判断**：
- 这是重大缺陷，不是普通疏漏
- 它会破坏用户对“我记得”的信任
- 会导致重复劳动、错误判断任务状态、错失后续准备窗口

**以后必须执行的规则**：
1. 只要聊天里完成了面试复盘 / 任务复盘 / 计划调整 / 关键决策，必须在当轮结束前写入文件
2. 回复用户时，明确说明“已写入哪个文件”；如果没写，必须明确说“这次只是临时讨论，尚未落盘”
3. 检索未命中文件时，不得直接推出“没有做过”，要优先判断“是否聊过但未落盘”
4. 心跳只负责巡检，不能被我拿来当记忆系统；关键内容要在发生当下完成沉淀
5. 对高价值信息，至少落到 `memory/daily/YYYY-MM-DD.md`；可复用内容再升级到长期主题文件或项目文件

**修复思路**：
- 短期：严格执行“重要讨论后立即落盘 + 明确告知文件位置”
- 中期：为高价值对话建立固定的复盘落盘动作
- 长期：把“会话成果沉淀”从可选动作升级为默认闭环

### 三层记忆闭环规则（2026-03-24）

**目标**：避免“聊过但没落盘”“检索不到就误判没做过”的系统性错误。

**三层闭环**：
1. **事件沉淀层（主闭环）**
   - 面试复盘、计划重大调整、偏好更新、关键决策、项目阶段结论，必须在当轮写入文件
   - 回复时必须明确说明“已写入哪个文件”；未写则明确说明“仅临时讨论，尚未落盘”
2. **心跳补漏层（兜底闭环）**
   - 心跳检查近期是否存在高价值内容未落盘、偏好未同步、任务状态与最新决策不一致
   - 低风险内容可自动补记到 daily；高风险内容主动提醒，不擅自写死
3. **调用前校验层（使用闭环）**
   - 在做计划、复盘、任务判断前，先核对最新偏好、任务状态、daily 记录与主题文件是否一致

**关键原则**：
- 心跳负责补漏，不替代事件当下沉淀
- 检索未命中文件时，不得直接推出“没做过”，要优先判断“是否聊过但未落盘”
- 记忆系统不是只做检索，而是覆盖形成→整合→调用的完整生命周期

### 长期信息写入规则（2026-03-24）

**场景**：向 `USER.md`、`AGENTS.md`、`memory/projects.md`、长期主题文件等写入高价值信息时，不能只凭印象直接追加。

**默认流程**：
1. **相关读取**：先读取目标文件相关 section，而不是盲目全文重读或完全不读
2. **查重检查**：用 grep 或局部检索检查是否已有同类表述
3. **合并写入**：优先 merge / replace / cancel old state，而不是无脑 append
4. **写后验证**：确认没有重复、没有冲突、状态表达一致

**适用范围**：
- 用户长期偏好
- 核心规则
- 项目/待办状态的重要变更
- 关键决策与长期约束

**核心原则**：
- 不要求每次全文通读，但必须做“相关读取 + 查重 + 合并写入 + 写后验证”
- 低风险日常日志可以轻量处理；高价值长期信息必须按上面四步执行

### 面试复盘的岗位校准与表达原则（2026-03-25）

**场景**：复盘当天面试时，若岗位标签识别错误（如把 AI Agent 工程师误按 FAE 评估），会直接导致判断偏差；即便思路正确，表达失控也会让有效信息只传达一半。

**以后必须执行的规则**：
1. **先校准岗位，再评价表现**：复盘前先确认真实岗位名称、JD主线、面试官关心的核心能力，避免用错评价框架
2. **区分两类问题**：
   - 方向错：答非所问、岗位主线跑偏
   - 表达差：方向是对的，但答案收不住、结论不前置、被面试官带着走
3. **面试回答先给结论，再给解释**：不要把“脑内搜索过程”直接说出来
4. **所有技术题尽量收束成三段式**：定义问题 → 核心矛盾 → 我的答案/做法
5. **真正的短板要说准**：当内容理解没问题时，不要误判成“能力不行”，而应明确指出是“表达配不上思考”

**可复用话术原则**：
- 固定元素交给约束，可变元素交给生成
- LLM负责理解/生成，工程负责约束/执行
- AI落地难点不只是prompt，而是可控性与稳定性
- 不幻想万能Agent，优先模块化拆分任务

**效果**：
- 今天成功解决了看似"无解"的网络搜索问题
- 发现了多个可用工具
- 建立了系统的诊断方法
