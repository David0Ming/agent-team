## 2. 他们通常会在写之前做“程序化查重 / 冲突检查”
这一步是我这次没做够的。

成熟做法一般会有这些机制：

### A. key-based memory
不是纯文本堆在一起，而是带 key
比如：
- `user.preference.planning_mode`
- `job.target_direction`
- `project.status.resume_java_version`

### B. append 之前先 merge
不是看到新信息就加一条，
而是判断：
- 这是新信息？
- 还是对旧信息的更新？
- 还是取消旧状态？

然后走：
- **replace**
- **merge**
- **mark obsolete**
而不是永远 append。

### C. conflict detection
如果新旧信息冲突，会触发：
- 覆盖旧值
- 留版本
- 要求确认

> **先改结构化状态，再同步到可读文档**
