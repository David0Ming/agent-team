# Shared Lessons

> **记录什么**：所有 Agent 都适用的通用工作规则。
> **为谁记录**：给全体 Agent 用。
> **不记录什么**：
> - 某个角色独有的经验（写到 lessons-djj / lessons-dming / lessons-david）
> - 项目专属知识（写到 projects/.../docs）
> - 工具配置与故障细节（写到 tools 或 knowledge）
> - 用户个人偏好（写到 USER.md）

## 通用工作方法

### 及时记录原则（2026-03-06）
**教训**：高价值内容不能只等会话结束时再记；异常结束会丢失信息。

**规则**：关键进展应在发生当下或当轮结束前写入文件。

### 不要过度创建新文件（2026-03-08）
**教训**：文件越分散，越难复用和维护。

**规则**：
1. 先找现有文件
2. 再判断是否真需要新文件
3. 优先补充已有主题，而不是新建临时文档

### 禁止创建版本文件（2026-03-14）
**教训**：`xxx-v1.md / xxx-v2.md` 会制造垃圾文件和认知负担。

**规则**：
1. 只保留一个主文件
2. 直接更新原文件
3. 历史版本交给 git 管理

### 问题诊断的系统性（2026-03-15）
**教训**：遇到问题不能只说“不行”，要系统诊断。

**规则**：
1. 列出相关工具/路径
2. 逐一测试并记录错误
3. 做根因分析
4. 尝试替代方案
5. 验证并记录最终可用方案

### 验证后再汇报（2026-03-09）
**教训**：不能只报告“完成了”，必须先验证结果存在且正确。

**规则**：
1. 创建文件 → 检查文件存在
2. 运行脚本 → 检查输出正确
3. 改配置 → 验证配置生效
4. 汇报时给证据，而不是只给结论

### 数据口径一致原则（2026-03-10）
**教训**：做分析时，最容易错的不是计算，而是口径混用。

**规则**：
1. 先区分累计 / 单日 / 平均
2. 对比时必须同口径
3. 趋势判断优先看时间序列，不拿单点替代趋势
4. 若用户要“昨天表现”，默认先确认他要的是单日增量还是累计结果
